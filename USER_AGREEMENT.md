## 第一章 系统架构与技术边界详述

### 1.1 分层架构的技术解剖与 YOLO 集成

AAM 采用模块化分层设计，各层的功能边界与技术约束如下，**特别标注 YOLO 在视觉感知层的作用与限制**：

**L0 Sensing 层（感知层）**  
本层负责与外部世界的原始交互。在 Android 平台上，它通过 **Android Debug Bridge (ADB) 协议**执行屏幕截图捕获与触控事件注入。技术边界声明：**本层不调用任何 Windows API 读取游戏进程内存（如 `ReadProcessMemory`），不注入 DLL 到游戏进程，不挂钩（Hook）系统或游戏的图形渲染 API（如 DirectX、OpenGL ES 拦截）**。共享内存（Shared Memory, SHM）仅用于 AAM Core (C++) 与 AAM Inference (Python) 进程间的内部数据交换，**绝不与《明日方舟》游戏进程共享内存空间**。

**L1 Perception 层（感知处理层）**  
本层是 AAM 的视觉处理核心，采用**双轨架构**：

- **传统 CV 轨道**：基于 **OpenCV 4.x**（Apache-2.0 许可证）进行模板匹配与 OCR（光学字符识别），用于识别固定 UI 元素（如费用数值、干员头像、按钮区域）；
- **YOLO 深度学习轨道**：基于 **Ultralytics YOLOv8/v9**（AGPL-3.0 许可证，与 AAM 许可证兼容）的 **ONNX 导出模型**，执行**实时目标检测**（Real-time Object Detection）。

**YOLO 技术细节与边界**：
- **输入**：屏幕截图（由 L0 层通过 ADB 捕获，16-33ms 延迟）；
- **检测类别**：敌方单位（按职业分类：领袖、重装、术师等）、我方干员部署位置（ bounding box 坐标）、技能就绪状态图标、特殊机制物体（如装置、障碍物）；
- **模型局限**：**纯视觉识别，不读取游戏内部实体 ID 或属性数值**；依赖 GPU 加速（CUDA/DirectML），在低端设备上可能存在 50-200ms 的推理延迟；存在**误检/漏检风险**（如相似敌人外观混淆、密集重叠单位漏检）；
- **卫戍协议特定**：YOLO 用于识别**随机出现的敌人类型**（如领袖单位、特训敌人）的**视觉特征**（模型通过训练数据识别特定敌人的像素模式），但**无法预测**敌人的波次生成逻辑或领袖技能释放时机。

**L2 Motor 层（执行层）**  
通过 **MaaFramework**（LGPL-3.0 许可证）提供的接口，将 L3/L5 层的决策转换为 ADB 命令序列，模拟人类手指的点击、滑动、长按等操作。操作执行引入随机延迟与坐标扰动（"人性化模拟"），但**不构成对游戏客户端的代码注入或协议层伪造**。

**L3 Tactical 层（战术决策层）**  
内置 **DSL（领域特定语言）编译器**，解析基于 **公开数据源**（PRTS Wiki、GamePress、官方攻略等社区维护的静态数据库）的关卡机制、敌人路线与属性数值，进行**预计算战术规划**。

**卫戍协议特定修正**：对于「卫戍协议」模式，L3 层**无法使用静态数据库**（因地图随机选择、领袖随机、盟约组合多变），此时系统**降级为 YOLO 视觉反应模式**：L1-YOLO 层实时检测战场敌情（敌人位置、生命值状态），L3 层基于**视觉反馈**而非**预计算**生成战术指令（如"检测到领袖进入射程则部署干员"）。这是一种**反应式（Reactive）**而非**预测式（Predictive）**的战术逻辑，**效率显著低于固定关卡的预计算模式**。

**L4 State 层（状态管理层）**  
维护游戏状态机的快照序列化，使用 **Protocol Buffers**（BSD-3-Clause 许可证）进行数据持久化与进程间通信（通过 **gRPC**，Apache-2.0 许可证）。本层数据**仅反映通过屏幕识别（含 YOLO 检测结果）推断的状态，而非游戏内部的真实内存状态**。

**L5 Strategy 层（战略决策层）**  
通过 **Bridge 层（gRPC / ZeroMQ, MPL-2.0 许可证）**接收 L4 层上传的结构化状态描述（含 YOLO 检测到的敌方单位列表），调用本地或远程的大语言模型（LLM）/视觉语言模型（VLM）API 生成高层战略建议。**在卫戍协议的多人模式下，L5 层仅接收本机阵地状态，无法获取其他玩家（同盟）的完整战场信息，可能导致协同策略失误**。

**Data 层（数据层）**  
存储干员属性、敌人图鉴、固定关卡路线等**公开可用数据**，使用 **nlohmann/json**（MIT 许可证）进行解析。**明确排除**对游戏客户端私有资源包（.ab 文件）的逆向解包或加密协议分析。

## 第二章 游戏特定性风险与技术迁移警告

### 2.1 跨游戏适用的风险梯度

**核心风险告知**：AAM 的 L0-L5 架构（含 **YOLO 视觉检测**）在技术上具有**跨游戏通用性**。若用户将此技术栈应用于含有**玩家间实时竞技（PvP）**机制的游戏（如第一人称射击游戏《Valorant》《CS2》、多人在线战术竞技游戏《英雄联盟》《DOTA2》），L1-YOLO 层的实时目标检测（自动瞄准、敌人位置追踪）与 L3 层的战术决策将立即构成**破坏竞技公平性的"人工智能外挂"**（如实时锁定敌方玩家头部坐标）。

然而，AAM 的当前实现针对《明日方舟》的特定技术环境进行了**约束性适配**，其风险等级差异如下：

| 技术维度 | 传统竞技游戏"外挂"（高风险） | AAM 在《明日方舟》中的实现（中低风险） |
|---------|------------------|-------------------------|
| **数据输入源** | 内存读取（DMA、Kernel Driver）、网络协议拦截（Man-in-the-Middle）、渲染层注入（Overlay Hook） | **仅通过 ADB 获取屏幕像素流**（YOLO 处理视觉帧），**零内存读取**，**零网络层注入** |
| **预测/检测基础** | 实时计算敌方玩家位置、技能冷却、经济状态（非公开数据，需破解加密） | **固定关卡**：基于 **已公开的敌人数据库** 预计算；**卫戍协议**：YOLO **实时检测**已出现的敌人视觉特征（仅反应，不预测生成逻辑） |
| **交互方式** | 内核级输入（驱动级鼠标移动、硬件级宏）、数据包伪造（瞬移、加速） | **仅模拟人类触控操作**（点击、滑动），通过 ADB 协议传输，**不涉及内核驱动或协议伪造** |
| **联机影响** | 直接破坏 PvP 竞技公平性，影响其他玩家体验 | 游戏核心为**单人/协作 PvE**；「卫戍协议」为**协作对抗 AI**，YOLO 仅辅助识别，**不控制其他玩家阵地**，且 L3 层**无法处理**多人协同的复杂策略（如联防阶段决策） |
| **经济系统** | 操控交易市场、自动套利、影响其他玩家经济利益 | **仅影响用户自身账号的本地资源消耗**，**不涉及玩家间交易操控** |

### 2.2 《明日方舟》游戏性质的特定性依赖

您明确理解并同意，AAM 当前被界定为"**高级自动化脚本**"而非"**竞技作弊工具**"，**严格依赖于**《明日方舟》的以下技术特性：

1. **半单机架构**：核心玩法（主线关卡、资源副本、剿灭作战、集成战略）基于**本地静态数据**运行，服务器仅作结果校验与数据持久化。L3 层的 DSL 编译器可解析**固定关卡配置**（如敌人行进路线、波次 timing、固定机制触发点）。

2. **卫戍协议的视觉辅助限制**：「卫戍协议」模式虽涉及多人协作与随机元素（随机领袖、随机地图、70秒决策时间限制），但其本质仍为**玩家协作对抗 AI（PvE）**。AAM 对此模式的支持**仅限于**：
   - **L1-YOLO 实时检测**：识别当前战场上的敌人类型（领袖、特训敌人）与位置；
   - **L3 反应式战术**：基于 YOLO 检测结果执行简单逻辑（如"费用充足且检测到领袖则部署银灰"）；
   - **明确排除**：**无法预测**随机地图布局（因地图在开局后随机选择，非固定数据库可预载）；**无法控制**其他玩家的阵地（同盟模式下仅能操作本机干员）；**无法处理**"联防阶段"的跨阵地协同（因涉及其他玩家状态，超出 YOLO 视觉范围）。

3. **公开数据依赖**：Data 层使用的干员属性数据库、敌人图鉴、固定关卡路线均来源于**社区公开维护的数据集**（PRTS Wiki、Arknights Terra Wiki、GamePress 等），**非通过逆向工程（Reverse Engineering）或反编译获取的加密资产**。

4. **非竞技性（Non-PvP）**：截至本协议生效日，《明日方舟》**不存在**玩家间直接对抗的排位赛、竞技场或经济对抗系统。「卫戍协议」的同盟模式虽有多人元素，但为**协作性质**，且 AAM **不保证**在多人环境下的决策有效性（如因网络延迟导致的 70 秒决策超时、因无法观察友方阵地导致的部署冲突）。

### 2.3 游戏机制变更的兼容性警告与风险升级

**关键风险提示**：若《明日方舟》在未来版本更新中引入以下机制，AAM 的技术性质可能发生**法律风险升级**，您有义务立即停止使用：

- **玩家间实时竞技（PvP）模式**：如排位竞技场、玩家对战玩法；
- **服务器端动态生成关卡**：如完全随机的地图布局、实时变化的敌人属性，使得 L3 层的静态数据库与 YOLO 的训练模型**同时失效**；
- **玩家间自由交易市场**：如允许玩家间交易虚拟货币或物品；
- **反作弊机制升级**：鹰角网络明确将"基于计算机视觉的自动化检测"（YOLO 实时识别）或"ADB 模拟触控"纳入行为检测范围，或开始封禁 ADB 调试接口的使用。

在上述情况下，继续使用 AAM 将**从"违反服务协议的自动化脚本"升级为"破坏游戏公平性的作弊工具"**，可能导致**更严厉的账号处罚**（永久封禁且不予申诉），甚至可能涉及**不正当竞争**的法律风险。

## 第四章 隐私保护、数据流与 AI 决策风险

### 4.1 本地优先的数据处理架构与 YOLO 隐私边界

AAM 默认采用**端侧计算优先**架构，数据流如下：

**L0-L4 层（C++ Core）**：  
- 屏幕截图（通过 ADB）仅在本地内存中处理，**不上传**至 Ethernos Studio 服务器；  
- **YOLO 推理**：在本地 GPU（NVIDIA CUDA 或 DirectML）上执行，**原始图像帧不离开本机**；  
- OCR 识别结果、YOLO 检测框坐标（bounding boxes）、游戏状态推断数据通过 **ZeroMQ (MPL-2.0)** 或 **gRPC (Apache-2.0)** 在本地进程间（Core ↔ Inference）传输，**不经过互联网**；  
- 操作日志（含 YOLO 检测结果摘要，如"检测到领袖：爱国者"）仅本地保存，默认 **7 日后自动不可逆清除**。

**L5 层（Python Inference）**：  
- 仅在您**主动启用** AI 决策功能时，结构化状态描述（含 YOLO 检测到的敌方列表、干员部署状态、费用数值，**不含原始屏幕截图**）通过 **HTTPS/TLS 1.3** 传输至您配置的第三方 LLM API（如 Moonshot AI、Anthropic Claude、OpenAI GPT 等）；  
- **强烈建议**在配置文件中启用 `privacy_mode: strict`，该模式将自动模糊处理用户 ID、充值记录、社交关系图谱等敏感元数据，**仅传输最小必要信息**（如仅发送"领袖已出现"而非具体坐标）。

### 4.2 第三方 AI 服务的隐私边界

若您启用 L5 层：  
- 您需**自行审阅**所配置 API 提供商的隐私政策（如 Moonshot AI 的《用户协议》与《隐私政策》），Ethernos Studio **不对**第三方服务的数据泄露、模型训练数据滥用或跨境数据传输合规性承担责任；  
- 建议优先使用**本地部署**的开源大模型（如 Llama 3、Qwen 2.5）以避免云端数据暴露；  
- 明确禁止将包含其他玩家个人可识别信息（PII）的游戏截图（如多人模式中的玩家昵称、UID）传输至第三方 AI 服务。

### 4.3 AI 决策与 YOLO 检测的局限性与风险披露

**L1-YOLO 检测的固有局限**：  
- **误检/漏检**：YOLO 模型基于固定训练数据集（历史活动截图），对于**首次出现的新敌人**（如新一期「卫戍协议」的新领袖）可能无法识别或错误分类（如将"萨卡兹百夫长"误检为"萨卡兹刀兵"）；  
- **遮挡与重叠**：在敌人密集或特效遮挡（如全屏技能动画）时，YOLO 可能漏检关键单位，导致 L3 层战术失效；  
- **分辨率依赖**：YOLO 模型对特定屏幕分辨率（如 1920x1080、2560x1440）训练，若您使用非标分辨率（如 21:9 超宽屏），检测精度可能下降；  
- **卫戍协议特定**：70 秒决策时间限制下的**实时推理压力**（若 GPU 性能不足，YOLO 推理可能占用 1-2 秒，压缩后续决策时间）；**多人模式下仅检测本机阵地**，无法观察友方部署导致的**协同失误**（如重复部署同类型干员）。

**L3 层战术局限（卫戍协议特定）**：  
- **反应式延迟**：对于「卫戍协议」，L3 层基于 YOLO 检测结果采取**事后反应**（敌人出现后才响应），而非固定关卡的**事前预部署**，**胜率显著低于人工操作**；  
- **无法处理随机盟约**：L3 层无法理解复杂的"盟约"组合效果（如特定阵营增益），可能做出次优决策；  
- **无法处理联防阶段**：当进入"联防阶段"（合并失败玩家剩余敌人）时，AAM **无法识别**友方阵地的干员配置（因仅捕获本机屏幕），可能导致部署冲突或资源浪费。

**L5 层 LLM/VLM 决策的固有局限**：  
- **幻觉（Hallucination）风险**：大语言模型可能错误解读 YOLO 检测结果（如将"领袖已进场"误判为"领袖已死亡"），生成完全错误的部署指令；  
- **上下文限制**：LLM 可能无法处理「卫戍协议」的长局记忆（如 20+ 回合的累积经济状态），导致后期决策失误。

**开发者明确声明**：**不对因 YOLO 误检、L3 反应式战术局限、或 L5 层 AI 决策失误导致的游戏内损失承担责任**，包括但不限于错误部署高价值干员、误消耗珍稀道具、关卡失败导致的理智值损失或活动奖励遗漏。
